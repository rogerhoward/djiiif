# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

`djiiif` is a small Django package that extends `ImageField` to expose IIIF Image API URLs. It is published to PyPI as `djiiif`; the package itself has no runtime application — it is consumed by Django projects that depend on it.

## Backwards compatibility

**Preserve backwards compatibility by default.** Because `djiiif` is a published library consumed by downstream Django projects, treat its public surface — the `IIIFField` / `IIIFFieldFile` / `IIIFObject` import paths, the `.iiif` attribute and its profile/`info`/`identifier` attributes, the `IIIF_PROFILES` config shapes (plain `dict` and callable), and the `{% iiif %}` template tag — as a contract that must not break without approval.

- **Do not introduce a breaking change unless the operator has explicitly called it out and approved the breakage.** If a task's cleanest implementation would break compatibility, stop and surface it rather than shipping the break silently — offer a backwards-compatible path first.
- **Any approved breakage must be documented for users** — a `CHANGELOG` / release-notes entry stating what changed, why, and the migration path — and updated in `README.md` and this file in the same commit.
- **Any approved breakage causes a MAJOR version bump** (semver: `X.0` → `(X+1).0`), never a minor/patch release. Bug fixes and new backwards-compatible features stay within the current major.

## Changelog

**Maintain `CHANGELOG.md` automatically.** It is the canonical, user-facing record of what changed — keep it current as part of the work, never as a follow-up.

- **Every change that affects users** — new features, behavior changes, bug fixes, deprecations, and (approved) breaks — gets an entry in `CHANGELOG.md` **in the same commit** as the code/doc change. Internal-only churn (refactors, test-only or CI-only changes, formatting) does not need an entry.
- Follow the existing format: [Keep a Changelog](https://keepachangelog.com/) headings (`Added` / `Changed` / `Fixed` / `Deprecated` / `Removed` / `Security`), newest first. Accumulate entries under a top **`## [Unreleased]`** section as you work.
- **Breaking changes** go under a **`Breaking`** heading in the entry and must state the migration path — this is the same entry the backwards-compatibility policy above requires, and it pairs with the MAJOR version bump.
- **At release time**, rename `## [Unreleased]` to the new version with the release date (`## [X.Y] - YYYY-MM-DD`), add a fresh empty `## [Unreleased]` above it, and update the compare/link references at the bottom. Keep the `CHANGELOG.md` version consistent with the release tag (which is the sole version source via setuptools-scm).

## Build & release

**Releases are automated.** Do not build or `twine upload` from a workstation.

**The version is derived from the git tag by `setuptools-scm`** — there is no hand-bumped version anywhere (no `setup.py`, no `__version__`). To cut a release:

1. Create a GitHub Release with tag `vX.Y` (e.g. `v0.24`) targeting the commit on `master`. `setuptools-scm` turns that tag into the package version (`v0.24` → `0.24`); a tagged commit builds a clean version, any other commit builds a dev version like `0.24.devN+g<hash>`. There is **no** tag-vs-version assertion to keep in sync (setuptools-scm made it unnecessary).
2. `.github/workflows/release.yml` then: checks out with `fetch-depth: 0` (so setuptools-scm can see tags), installs `.[dev]`, runs the test suite, builds sdist + wheel via `python -m build`, runs `twine check`, and publishes to PyPI via OIDC trusted publishing (no API token in secrets).

**One-time PyPI setup required:** the project must have a Trusted Publisher configured at https://pypi.org/manage/project/djiiif/settings/publishing/ pointing at `rogerhoward/djiiif`, workflow `release.yml`, environment `pypi`. The matching GitHub Environment `pypi` can also gate the publish job with manual approval / branch restrictions.

All project metadata and the build/docs/test toolchain live in **`pyproject.toml`** (PEP 621): runtime dependency `Django` under `[project] dependencies`; the DRF extra under `[project.optional-dependencies] drf`; the docs toolchain (`sphinx`, `furo`, `myst-parser`, `sphinx-copybutton`, `djangorestframework`) under the `docs` extra (used by Read the Docs); and the full dev toolchain (`pytest`, `pytest-django`, `pytest-cov`, `djangorestframework`, `build`, `twine`, plus `djiiif[docs]`) under `dev`. There is no `setup.py`, `Pipfile`, `MANIFEST.in`, or `requirements.txt` — they were removed in the 0.24 packaging modernization.

## Tests

```bash
pip install -e ".[dev]"   # one-time setup; installs Django + the full test/build/docs toolchain (incl. DRF for the serializer tests)
pytest                                                    # runs all tests + coverage gate
pytest tests/test_iiif_object.py::test_dict_profile_builds_url   # single test
```

Tests live in `tests/`. `tests/conftest.py` configures a minimal in-memory Django (no DB) via `settings.configure(...)` — there is no `manage.py` and no `DJANGO_SETTINGS_MODULE`. Pytest config (testpaths, coverage flags, fail-under threshold) is in `pyproject.toml`.

CI runs the same `pytest` invocation across Python 3.10–3.13 via `.github/workflows/test.yml`.

**Coverage gate: 90% (`--cov-fail-under=90` in `pyproject.toml`).** Any change that drops coverage below the gate must either add tests or — if the line is genuinely defensive/unreachable — raise the gate intentionally rather than silently lowering it.

## Architecture

The core is `djiiif/__init__.py`; the other modules are thin, optional add-ons around it.

- `djiiif/__init__.py` — defines `IIIFField` (subclass of `ImageField`) whose `attr_class` is `IIIFFieldFile` (subclass of `ImageFieldFile`). Accessing `.iiif` on a field file returns a freshly constructed `IIIFObject`, which reads `settings.IIIF_PROFILES` and sets one attribute per profile name holding the assembled IIIF URL, plus an `info` attribute for the IIIF `info.json` **URL** and an `identifier` attribute for the plain `host/identifier` URL (used for openseadragon-style integrations). Empty/unset fields — **and an empty `IIIF_PROFILES`** — produce empty-string URLs without raising (the "safe for empty fields" behavior, extended in 0.24 to the no-profiles case).
  - **Module-level builders** (all public, all reused by the serving view): `encode_identifier(name)` percent-encodes the identifier segment; `resolve_profile(profile, parent)` normalizes any profile shape to a spec `dict`; `image_url(spec, identifier)` assembles the Image API URL; `build_info_document(...)`, `build_manifest(...)`, `build_multi_manifest(...)`, and `build_collection(...)` construct the JSON documents; `_api_version`/`_compliance_level` resolve the version/level settings. `IIIF_CONTEXTS` maps Image API version → `@context` URI; `PRESENTATION_CONTEXT` is the Presentation 3.0 context; `PROFILE_KEYS` is the required-key tuple. `_manifest_uri`/`_canvas_uri` are the single derivation point for the synthetic `{id}/manifest` and `{id}/canvas/1` URIs, shared by `build_manifest` and `IIIFObject.content_state` so they cannot drift.
  - **info.json enrichment** (declarative pass-through, opt-in): `InfoExtras` (frozen dataclass, 10 optional fields) / `resolve_info(parent)` (dict/`InfoExtras`/callable→either/`None`, mirrors `resolve_auth`; `parent` is the `IIIFFieldFile` on model paths and the decoded storage **name** `str` on view paths). `INFO_KEYS` maps canonical snake_case → emitted spec camelCase (`max_width`→`maxWidth`, …); `_normalize_info_keys` accepts either spelling and rejects unknown keys and snake/camel duplicates (`ImproperlyConfigured`). `build_info_document(..., extras=)` applies them via `_apply_info_extras`: v3 emits all present keys in `INFO_KEYS` order; v2 emits only `sizes`/`tiles` and raises on v3-only keys. Threaded through `IIIFObject.info_document` and `serve_info_json`. Unset ⇒ byte-identical. djiiif only *declares* — it never probes the image server.
  - **Web Annotations + Content Search 2.0** (shared `Annotation`, opt-in views): one frozen `Annotation` dataclass (fields `text`/`motivation="supplementing"`/`xywh`/`language`/`format="text/plain"`/`id`/`canvas_id` + search-snippet `exact`/`before`/`after`) is the single hit/annotation type; `resolve_annotation(entry)` normalizes an `Annotation` or plain dict (defaults from `_ANNOTATION_DEFAULTS`; bad type → `ImproperlyConfigured`). `build_annotation`/`build_annotation_page` build a W3C `AnnotationPage` over `ANNOTATION_CONTEXT` (`[anno, presentation3]`; `_annotation_body` wraps a string as `TextualBody` or passes a preformed body dict through; a search hit's own `canvas_id` supplies the target when the builder is passed `canvas_id=None`). `build_search_service` is the `SearchService2` block; `build_search_response(search_url, q, hits, *, ignored=())` is the Search 2.0 `AnnotationPage` (single `SEARCH_CONTEXT`, embedded hit annotations in `items`, a `contextualizing` match block of `SpecificResource` + `TextQuoteSelector` where `exact` defaults to `q`). No `IIIFObject` surface — advertisement is view-only.
  - **Change Discovery API 1.0** (public builders, opt-in views): `Activity` (frozen dataclass — `object_id`/`end_time`/`type="Update"`/`object_type="Manifest"`) is the typed activity-source entry; `resolve_activity(entry)` normalizes an `Activity` or plain dict to a dict (defaults applied; bad type → `ImproperlyConfigured`), mirroring `resolve_profile`. `build_activity(object_id, end_time, *, activity_type, object_type)`, `build_ordered_collection(id_url, total, first_url, last_url)`, and `build_collection_page(id_url, collection_url, activities, *, prev_url, next_url, start_index)` are pure builders over the `DISCOVERY_CONTEXT` (`[discovery/1, activitystreams]` array — brief-specified; the spec's own examples use the single discovery string, which re-exports AS2). Activities carry no `@context` (always embedded in a page); ordering is the source's contract (not re-sorted).
  - **Presentation enrichment** (0.24+): `build_multi_manifest(id_url, images, *, label, ..., **descriptors)` is the one canvas-assembly path (shared `_build_canvas`; each image spec normalized by `_normalize_image_spec` — a `(service_id_url, width, height)` tuple or a dict with an optional per-canvas `label`); `build_manifest` is a thin one-image wrapper over it, so single-image output is byte-identical. Both accept optional descriptive-property kwargs (`DESCRIPTOR_KEYS` = `metadata`/`rights`/`required_statement`/`summary`/`thumbnail`/`nav_date`), validated by `_reject_unknown_descriptors` and assembled by `_descriptive_properties` (coercion helpers `_metadata_pairs`/`_label_value`/`_thumbnail`; label-ish values go through `_language_map`). `resolve_manifest_descriptors(parent)` reads `settings.IIIF_MANIFEST_DESCRIPTORS` (a `dict`, a callable → `dict`/`None`, or unset; mirrors `resolve_auth`) and is threaded into `IIIFObject.manifest`. `build_collection(id_url, items, *, label, **descriptors)` emits a `Collection` of manifest *references* (`_collection_item` coerces `(manifest_url, label[, thumbnail])` tuples or passes preformed dicts through). Auth applies to every image body in a multi-manifest. Both manifest builders also take `nav_place` (a resolved GeoJSON `FeatureCollection` from `djiiif.geo.resolve_navplace`): when set, the manifest emits `navPlace` (Features get synthesized `{manifest_id}/navplace/feature/{n}` ids via `_navplace_with_feature_ids`, not mutating caller data) and `@context` becomes the two-element `NAVPLACE_CONTEXT` array (extension context first). Unset ⇒ string context, no `navPlace` (byte-identical).
  - **Content State API 1.0 helpers** (public, no new settings/deps): `encode_content_state(state)` / `decode_content_state(encoded)` are the spec §6 base64url pair — note the encode step percent-encodes the JSON via `quote(..., safe="!~*'()")` to match JavaScript `encodeURIComponent` *before* base64url + padding-strip, matching the spec's worked example byte-for-byte (the brief's prose omits this step; the code and the interop test are authoritative). `build_content_state(manifest_id, *, canvas_id=None, xywh=None)` assembles the spec's simplified target dict (Manifest, or Canvas with `partOf` + optional `#xywh=` region; `_format_xywh` accepts a string verbatim or a 4-tuple). `IIIFObject.content_state(*, xywh=None, encoded=True)` builds the state for this image's own canvas (I/O-free; `""`/`None` for an empty field).
  - `Profile` (frozen dataclass, added 0.24) is the typed, opt-in profile shape with 3.0 defaults (`size="max"`) and `mirror`/`upscale` flags that fold into the `!`-rotation / `^`-size prefixes via `.as_spec()`.
  - `IIIFObject.info_document` and `IIIFObject.manifest` (both `cached_property`, 0.24) are distinct from the eager URL attributes: they return **documents** (`dict`, or `None` for an empty field) built from the image's own `width`/`height`. They are the *only* attributes that read the file from storage, so constructing an `IIIFObject` stays I/O-free (the constructor stores `self._parent` for them). Both honor `IIIF_IMAGE_API_VERSION` (default `3`; `2` switches to 2.x shapes) and `IIIF_COMPLIANCE_LEVEL` (default `"level2"`); an unknown version raises `ImproperlyConfigured`.
  - `IIIFObject.as_dict(include_meta=False)` returns `{profile_name: url}` for every configured profile (optionally adding `info`/`identifier`), using the `self._profile_names` captured at construction. Backs the DRF serializer and template iteration.
  - **Auth Flow 2.0** (metadata-only, opt-in): `ProbeService` / `AccessService` / `TokenService` / `LogoutService` frozen dataclasses build the nested `AuthProbeService2 → AuthAccessService2 → AuthAccessTokenService2 / AuthLogoutService2` block via `.as_dict()` (label-ish fields coerced to IIIF language maps by `_language_map`). `resolve_auth(parent)` reads `settings.IIIF_AUTH` (a `ProbeService`, `dict`, callable → either/`None`, or unset) → a probe-service `dict` or `None`, mirroring `resolve_profile`. When non-`None`, the block is threaded (via the `auth=` kwarg) into `build_info_document` (top-level `service`) and `build_manifest` (the image body's `service`). Auth 2.0 is v3-only: `_require_auth_v3` raises `ImproperlyConfigured` if `IIIF_AUTH` is set while `IIIF_IMAGE_API_VERSION = 2`. djiiif only *describes* the services; the image server (iiiris, the preferred IIIF server) implements/enforces them. Mode B (servers without native Auth 2.0) is out of scope — see `briefs/AUTH-API.md`.
- `djiiif/views.py` + `djiiif/urls.py` (0.24) — optional drop-in `serve_info_json` / `serve_manifest` / `serve_collection` views and their URLconf. The first two map an encoded identifier back to a stored image via `default_storage` (shared `_load_dimensions` helper reads dimensions with `get_image_dimensions`), and return `build_info_document(...)` / `build_manifest(...)` as `application/ld+json` with a `*` CORS header (shared `_ld_json` helper); the document `id` comes from the request URL. `serve_collection` (at the fixed `/iiif/collection` route, listed before the greedy identifier patterns) reads no storage — it renders `settings.IIIF_COLLECTION_SOURCE` (callable/iterable of member entries; unset ⇒ `Http404`) with the optional `IIIF_COLLECTION_LABEL` via `build_collection`. `serve_activity_collection` (`/iiif/activity/collection`) and `serve_activity_page` (`/iiif/activity/page/<int:page>`) serve the Change Discovery stream, also storage-free: the shared `_activity_paginator` resolves `settings.IIIF_ACTIVITY_SOURCE` (dotted-path string via `import_string`, callable, or direct iterable; unset ⇒ `Http404`) — querysets/lists (anything with `__getitem__`) paginate in place, bare generators are materialized — into a `Paginator` sized by `IIIF_ACTIVITY_PAGE_SIZE` (default 100); an out-of-range page → `Http404`. `serve_annotation_page` (`/iiif/<identifier>/annotations/1`) and `serve_search` (`/iiif/<identifier>/search?q=`) are also storage-free: `_resolve_backend` reads `settings.IIIF_ANNOTATIONS_BACKEND` / `IIIF_SEARCH_BACKEND` (dotted-path string via `import_string`, or callable). The annotation page and (in the fallback) the search hits target `{id_url}/canvas/1` (`_canvas_id_for`), matching `serve_manifest`'s canvas. `serve_search` uses the dedicated search backend when set, else `_substring_hits` filters the annotations backend case-insensitively (`_search_available`); empty/missing `q` ⇒ empty page; neither backend ⇒ `Http404`. `serve_manifest` injects the canvas `annotations` reference (annotations backend set) and a manifest-level `SearchService2` (`_search_available`) — the only place these are emitted (`IIIFObject.manifest` stays unchanged). Included via `path("iiif/", include("djiiif.urls"))`; uses default storage only.
- `djiiif/serializers.py` (0.24) — **optional** DRF field `IIIFSerializerField` (read-only; `to_representation` returns `iiif.as_dict(...)`). Imports `rest_framework`, so it is an extra (`pip install djiiif[drf]`, declared under `[project.optional-dependencies] drf` in `pyproject.toml`) and is **never imported by `djiiif/__init__.py`** — the core package has no DRF dependency. It is omitted from the coverage source (its tests `importorskip` DRF and CI installs the `drf` extra to run them).
- `djiiif/geo.py` — **optional** GeoDjango bridge for the navPlace extension (`resolve_navplace`). **Never imported by `djiiif/__init__.py`** (imported lazily inside `IIIFObject.manifest`, explicitly by `views.py`) and **never imports `django.contrib.gis`** — a GEOS geometry is recognized by duck-typing its `.geojson`/`.srid`, so it stays fully unit-testable (and 100%-covered) without GDAL/GEOS. `resolve_navplace(parent)` reads `settings.IIIF_NAVPLACE` (dotted-path str / callable / direct value → GeoJSON dict / GEOS geometry / `(geometry, label)` / `None`), normalizes to a fresh GeoJSON `FeatureCollection` (`_geometry_to_dict` + `_to_feature_collection`; `_language_map` for the label), and raises `ImproperlyConfigured` on a bad type, an unexpected GeoJSON `type`, or a GEOS SRID set and ≠ 4326. Reuses `_language_map` from the core. A real-GEOS test exists but `pytest.skip`s when the GDAL native library is absent (importing `django.contrib.gis.geos` raises `ImproperlyConfigured`, not `ImportError`, so `importorskip` alone is insufficient).
- `djiiif/checks.py` + `djiiif/apps.py` (0.24) — `DjiiifConfig.ready()` registers `check_iiif_profiles`, a system check that validates `IIIF_PROFILES` at startup (ids `djiiif.W001`/`E001`/`E002`/`E003`). Callable and `Profile` entries pass unchecked (a callable's shape is only knowable at call time).
- `djiiif/templatetags/iiiftags.py` — registers the `{% iiif imagefield 'profile' %}` template tag (delegates to `getattr(imagefield.iiif, profile)`) and the `{% iiif_content_state imagefield [xywh='x,y,w,h'] %}` tag (delegates to `imagefield.iiif.content_state(xywh=...)`). Both raise `NotAnIIIFField` for a non-IIIF first argument.
- `djiiif/templatetags/__init__.py` — **dead code**: contains an out-of-sync duplicate of `IIIFObject` / `IIIFField` / `urljoin`. Nothing imports it. Treat `djiiif/__init__.py` as the source of truth; this file should be emptied as part of cleanup.

Profiles in `settings.IIIF_PROFILES` may be a `dict` with keys `host, region, size, rotation, quality, format`, a `Profile` instance, or a callable receiving the `IIIFFieldFile` and returning either a `dict` or a `Profile` — callables enable per-image logic (e.g. square crop using `parent.width`/`parent.height`). The identifier segment is the field's `name` fully percent-encoded via `encode_identifier` (`/` → `%2F`, plus the rest of the reserved set); as of 0.24 this replaces the old `/`-only `str.replace`, so names containing spaces/`?`/`#`/etc. now encode correctly (a URL change only for such names).

## Keeping docs in sync

Documentation in this repo must stay current with the code. When you change behavior, config, commands, or layout, update the affected docs in the **same commit** — not as a follow-up.

- `README.md` is the **single source of truth** for the readme — there is no `README.rst` (it was removed to eliminate the two-file drift). Do not recreate a second readme file; if a reStructuredText copy is ever needed (e.g. for a tool that requires it), generate it from `README.md` rather than hand-maintaining a duplicate.
- `CLAUDE.md` (this file) is the contract for future Claude sessions — if you change build/test commands, the public API surface, the testing setup, the coverage gate, or the file layout, update CLAUDE.md so the description matches reality. Stale guidance here is worse than no guidance.

### The documentation site (`docs/`, Read the Docs)

The Sphinx site under `docs/source/` is the **comprehensive, user-facing documentation**, published on Read the Docs (`.readthedocs.yaml` at the repo root; built via `docs/source/conf.py`, dependencies from the `docs` extra in `pyproject.toml`). It is the deep reference; `README.md` is the concise GitHub landing/quickstart. Both must stay current — treat the docs as first-class, not an afterthought.

**Maintenance policy — keep the docs comprehensive *and* user-friendly:**

- **Every user-facing change updates the docs in the same commit** — a new setting, view, builder, helper, template tag, or behavior change gets its narrative page updated (or a new page added) *and* the {settings/API references refreshed} alongside the code, exactly like `README.md` and the `CHANGELOG`. A PR that adds a feature without docs is incomplete.
- **Be comprehensive:** every public setting appears in `docs/source/settings.md`; every public class/function appears in the `api.rst` autodoc reference (add an `autoclass`/`autofunction` entry when you add one). The API reference is generated from the Google-style docstrings, so writing good docstrings *is* writing that part of the docs.
- **Be user-friendly:** lead with a runnable example on every page. Prefer showing a real `settings.py` / `models.py` / view / template snippet over prose. Keep the "opt-in, backwards-compatible, no output change until configured" framing. Use `pycon`/`code-block` examples with realistic values (encoded identifiers, real hosts).
- **Structure:** pages live in `docs/source/*.md` (MyST Markdown; `api.rst` is reStructuredText for autodoc). The `index.md` toctree is grouped Getting started / Guides / Reference — add new feature guides under **Guides**, keep **Reference** to settings/API/changelog. `changelog.md` `{include}`s the root `CHANGELOG.md` (don't duplicate it); the version is read from package metadata (never hand-edit it in `conf.py`).
- **The build must stay warning-clean.** Read the Docs runs with `fail_on_warning: true`. Before committing docs changes, run `python -m sphinx -W -b html docs/source docs/build/html` (install with `pip install -e ".[docs]"`) and fix every warning — a broken `{doc}`/`{class}` cross-reference or missing autodoc target fails the RTD build.
- **Don't duplicate narratives that drift.** Facts with one home stay there: the changelog is included, not copied; the API reference is autodoc, not hand-written. New prose goes in the guide pages.

## Python conventions

- Target Python 3.10+ (`requires-python = ">=3.10"` in `pyproject.toml`). Keep this and the trove classifiers in sync when bumping the floor.
- Use `from __future__ import annotations` is unnecessary on 3.10+; prefer native PEP 604 (`X | None`) and PEP 585 (`list[str]`) syntax.
- Add type hints to all new/edited public functions, methods, and class attributes. Keep them precise — `Any` is a code smell, not a default.
- Prefer f-strings over `.format()` / `%`. The existing `'{}.{}'.format(...)` calls in `djiiif/__init__.py` and `iiiftags.py` are fine to leave alone but should be migrated when touched.
- **Docstrings & comments use [Google style](https://google.github.io/styleguide/pyguide.html#38-comments-and-docstrings)** (`Args:` / `Returns:` / `Raises:` sections, one-line summary first). Every public module, class, and function gets a docstring; comments explain *why*, not *what*. **Whenever you touch a function, method, class, or module, create or update its docstring and any nearby comments so they match the new behavior in the same edit** — stale or missing docs on code you changed is not acceptable. Don't leave a wall of prose: keep summaries tight and only add `Args:`/`Returns:`/`Raises:` sections where they carry real information.
- Keep `IIIFField` / `IIIFFieldFile` / `IIIFObject` import paths stable — they are part of the public API consumed by downstream Django projects.

## Required test coverage

Every change to `djiiif/` must keep the suite green and the coverage gate satisfied. When adding behavior, add tests alongside it. The suite must always cover:

- Both `IIIF_PROFILES` shapes: plain `dict` profile and callable profile.
- The empty-/`None`-`name` path (must return `""` for every profile attr, for `info`, and for `identifier`) — this is the regression guarded by the 0.19 / 0.20 commits.
- Identifier encoding: a field name containing `/` must appear as `%2F`, and other reserved characters (space, `?`, `#`, …) must be percent-encoded too.
- The `{% iiif %}` template tag's happy path and its `NotAnIIIFField` error path, plus the `{% iiif_content_state %}` tag's happy path and `NotAnIIIFField` path.
- **Content State**: `encode_content_state`/`decode_content_state` round-trip (padding stripped; bare-URI-string payload survives) and byte-for-byte agreement with the Content State API 1.0 §6 worked example; `build_content_state` for the Manifest-only, Canvas + `partOf`, and `xywh` (string == tuple) shapes; `IIIFObject.content_state` deriving the same URIs `manifest` emits, the region form, the encoded↔dict equivalence, and `""`/`None` for an empty field.
- `info_document`: the default (v3) and `IIIF_IMAGE_API_VERSION=2` document shapes, the `IIIF_COMPLIANCE_LEVEL` override, `None` for an empty field, the `ImproperlyConfigured` unknown-version path, and that `.info` (the URL) is unchanged alongside it.
- **navPlace**: `resolve_navplace` for unset/`None`, bare geometry, Feature, FeatureCollection (fresh-copy passthrough), `(geom, label)` pair, GEOS geometry (duck-typed stub + real-GEOS `pytest.skip`-on-missing-GDAL), direct non-callable value, dotted-path, and the bad-type / unexpected-GeoJSON-type / SRID≠4326 rejections; manifest emission with `navPlace` + two-element `@context` when set and **absent with string context when unset** (regression pin), feature-id synthesis (supplied ids preserved, caller dict not mutated); `serve_manifest` threading with the decoded-name callable contract; and that importing `djiiif` pulls in neither `djiiif.geo` nor `django.contrib.gis`.
- **info.json enrichment**: each `IIIF_INFO` key emitted with correct spec casing/placement/order at v3; `sizes`/`tiles` at v2 and a v3-only key at v2 raising `ImproperlyConfigured`; camelCase and snake_case both accepted with conflicting duplicates rejected; `resolve_info` for dict/callable/`InfoExtras`/unset plus the bad-type path; **unset ⇒ byte-identical** documents (both versions, key-order pin); composition with `IIIF_AUTH`; and `serve_info_json` threading with the decoded-name (`str`) callable contract.
- `Profile`: the 3.0 defaults, the `mirror`/`upscale` prefix folding (including the no-double-prefix guards), and that a `Profile` works as an `IIIF_PROFILES` entry; `resolve_profile` for dict/`Profile`/callable inputs plus its two `ImproperlyConfigured` rejection paths.

## Packaging

Packaging was modernized in 0.24 — the notes below describe the **current** state (the old `setup.py` + `build.sh` + `MANIFEST.in` + `Pipfile` are gone).

- **PEP 621 metadata** lives entirely in `pyproject.toml` under `[project]` (setuptools backend). `[tool.setuptools.packages.find] include = ["djiiif*"]` ships only the package, never `tests/`.
- **`setuptools-scm`** derives the version from the git tag (`[tool.setuptools_scm]`); `dynamic = ["version"]`. Never add a hand-written version. Any tooling that reads a version must use `importlib.metadata.version("djiiif")` or the git tag, not a source constant.
- **Build** with `python -m build` (never `setup.py`). The sdist includes README/LICENSE automatically via setuptools-scm's git file finder, so there is no `MANIFEST.in`.
- **Dependencies**: runtime `Django` under `[project] dependencies`; optional extras `drf` and `dev` under `[project.optional-dependencies]`. Local/CI dev uses `pip install -e ".[dev]"`.
- Because there is no lock file, Dependabot's pip ecosystem now tracks only the direct deps declared here (this deliberately ended the transitive `Pipfile.lock` alert churn).
